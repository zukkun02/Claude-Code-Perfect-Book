# データを取り続けるための設計メモ（LINE 固有編 含む）

> 課題：RAG は集まり続ける生データがあって初めて生きる。**普段の生活動線にデータ取得の入口を埋め込む**ための設計集。
> 関連：[Slack起点RAG実装メモ.md](./Slack起点RAG実装メモ.md) / [エンべディングログ.md](./エンべディングログ.md) / [0425レポート.md](./0425レポート.md)

## 大原則：データが集まらない最大の理由は「摩擦」

「書く場所が遠い」「あとで整理しよう」と思った時点で人間は書きません。
**普段使っている窓口の中で完結させる**のが、データを取り続ける唯一のコツ。

---

## 1. データソースの分類（汎用）

| カテゴリ | 具体例 | 摩擦 | みずきの場合の現状 |
|---|---|---|---|
| **音声・口頭** | iPhoneボイスメモ、移動中の独り言、Zoom録画 | 低 | ◯ 既に運用中 |
| **テキスト発信** | X投稿、ブログ、note | 中 | △ X 過去ログ未取り込み |
| **チャット** | **LINE**、Slack、Discord | 低 | ✕ 未連携（最大の伸びしろ） |
| **メール** | Gmail、受信箱、自分宛メモ | 低 | △ Gmail MCP は接続済み |
| **読書・視聴** | Kindleハイライト、Readwise、YouTube視聴履歴 | 中 | ✕ |
| **行動ログ** | GitHub commit、Notion編集、カレンダー | 自動 | △ |

→ 音声以外がほぼ手付かず。**「LINE を取り込む」が次の一打として圧倒的にコスパ高い**。

---

## 2. LINE 固有の取り込み方（4 パターン）

LINE は閉じたエコシステムなので、データを外に出す方法は限定的。**現実的なのは 4 つ**。

### A. 自分専用 LINE Bot を作る（最強・推奨）

```
あなた → LINE で Bot にメッセージ送信
   ↓ Webhook
Cloud Function (GAS / Vercel)
   ↓ Notion API で追加
Notion DB（思考メモ）
   ↓ 後で同期ジョブ
Supabase pgvector
```

**メリット**
- LINE は **24h 起動してる窓口** なので、思いつき即書ける
- 画像・音声も送れる（音声送信 → 文字起こし → Notion 自動化が組める）
- 友達追加した自分の Bot が "外付け脳" になる
- LINE Messaging API は **月 200 通まで無料**（自分用なら十分）

**最短手順**
1. [LINE Developers](https://developers.line.biz) でチャネル作成（10 分）
2. Messaging API → Webhook 有効化
3. Webhook 先を Vercel / GAS にして「メッセージ受信 → Notion 投入」を 30 行書く
4. QR で自分の LINE に Bot を友達追加 → 完成

### B. 「Keep メモ」+ 週 1 エクスポート（手動だが超低コスト）

LINE 内に「**Keep メモ**」（自分専用トーク）がある。日々のメモはここに投げる。

- 週 1 でトーク履歴を `.txt` でエクスポート → メール → Gmail → Notion
- Gmail MCP はもう繋がっているので、**Claude Code に「Gmail 受信箱から LINE エクスポート読んで Notion DB に投入して」** と頼める
- Bot を作る初動コスト 0、運用は週 5 分

### C. iPhone ショートカット連携

- ショートカット「LINE で自分にメッセージ送る」と「Notion API 叩く」を二段で組む
- ホーム画面 / Apple Watch / "Hey Siri" から起動可能
- 音声 → ショートカット → 文字起こし → Notion 自動投入、まで一気通貫

### D. LINE WORKS API（法人向け、個人なら不要）

社内連絡が LINE WORKS なら API でトーク取得可能。個人運用ならフィットしない。

### 比較

| 方式 | 初期工数 | 日次運用負荷 | 取り込めるもの | おすすめ度 |
|---|---|---|---|:---:|
| **A. 自分専用 Bot** | 1 日 | ほぼ 0（書くだけ） | テキスト・画像・音声 | ★★★★★ |
| B. Keep メモ + 週1 export | 0 | 5 分 / 週 | テキストのみ | ★★★ |
| C. ショートカット | 半日 | 0 | テキスト・音声 | ★★★ |
| D. WORKS API | 1 日 | 0 | テキスト | ★（個人は不要） |

## 3. 自分専用 LINE Bot 最小実装（イメージ）

GAS（Google Apps Script）なら無料でホストできて 30 分で建つ。

```javascript
// Webhook エンドポイント
function doPost(e) {
  const event = JSON.parse(e.postData.contents).events[0];
  if (event.type !== 'message' || event.message.type !== 'text') return;

  const text = event.message.text;
  const userId = event.source.userId;
  const ts = new Date(event.timestamp).toISOString();

  // Notion DB に追加
  UrlFetchApp.fetch('https://api.notion.com/v1/pages', {
    method: 'post',
    headers: {
      'Authorization': 'Bearer ' + NOTION_TOKEN,
      'Notion-Version': '2022-06-28',
      'Content-Type': 'application/json'
    },
    payload: JSON.stringify({
      parent: { database_id: INBOX_DB_ID },
      properties: {
        '名前':       { title: [{ text: { content: text.slice(0, 80) } }] },
        'テキスト':   { rich_text: [{ text: { content: text } }] },
        '作成日時':   { date: { start: ts } },
        'ソース':     { select: { name: 'LINE' } },
        'ステータス': { status: { name: '未着手' } }
      }
    })
  });

  return ContentService.createTextOutput('OK');
}
```

→ デプロイ URL を LINE Developers の Webhook 欄に貼る、それだけ。

## 4. 取得した後の "整え" を自動化する

データを集めても**生のまま**だと検索精度が落ちる。LLM に整えさせる中間レイヤーを挟む。

```
LINE Bot
  ↓ raw text
Notion DB「Inbox」（無加工で蓄積）
  ↓ 1 日 1 回のエージェント
Claude/GPT で:
  - 要約を生成（埋め込み対象に）
  - カテゴリ・タグを推定（メタデータに）
  - 重複 / 雑談を除外
  ↓
Notion DB「整理済み」（embed 対象 + 構造化メタ付き）
  ↓ 同期ジョブ
Supabase pgvector
```

→ 「埋め込み対象 vs メタデータ」の分離は、**この自動整え工程で確定する**のが理想。
→ 雑に書く LINE の自由さと、検索しやすい構造化を両立できる。

## 5. 「取り続ける」ための出口設計

入力の話だけしていると続かない。**集めたデータが何かに使われる体験**を週 1 で作ると、入力モチベが切れない。

| 出口 | やること | 効果 |
|---|---|---|
| **週 1 ハイライト** | "今週のあなたのテーマ Top 3" を Slack/LINE に通知 | 「集めたデータが意味を持つ」可視化 |
| **ポスト案 自動提案** | 毎日 9:00 に X 投稿案 3 本を Slack に投下 | 即時に発信に使える |
| **過去の自分検索** | "1 年前の今日、何考えてた？" を週 1 通知 | 思考の蓄積感が出る |
| **Reflection** | 集めたログから "似たことを 3 回以上言った" 主張を抽出 | 自分の核がはっきりする |

→ 入口（LINE Bot）と出口（Slack / LINE への日次通知）を**両方つなぐ**と、フライホイールが回り始める。

## 6. みずきの場合の推奨ロードマップ

| Week | やること | 効果 |
|---|---|---|
| **W1** | 自分専用 LINE Bot を作って `Inbox DB` に投入 | データ流入チャネル開通 |
| **W2** | Inbox DB → 整理エージェント（Claude / GPT で要約 + タグ付け） | メタデータ整備 |
| **W3** | 整理済み DB → Supabase pgvector に同期 | 自前検索基盤完成 |
| **W4** | Slack / LINE に「日次ポスト案」を返す Bot を構築 | 出口完成・フライホイール起動 |

→ 1 ヶ月で**入口（LINE）→ 整理 → 検索基盤 → 出口（Slack/LINE）**の循環が完成。

## ひとこと要約

> **LINE は最強の入口になる**。普段使ってる窓口に Bot を置くだけで、思考が手放されずに溜まる。
> **取りっぱなしは死蔵**。整理エージェント（要約・タグ付け）を 1 段挟むのが必須。
> **出口を作ると入口が続く**。週 1 でも "使われている" 実感を作れば、データは自然に増える。

> **claudeからの補足**
> - LINE Bot のメッセージ受信は **完全プライベート**（公式アカウント運営者＝あなたしか見ない）。心配無用。
> - "音声を LINE に送る → Whisper で文字起こし → Notion" まで自動化すると、**歩きながら独り言を Bot に投げて思考が蓄積する**最強体験になる。Whisper API で 1 分 $0.006 なので月数百円。
> - 既に Notion / Gmail / Calendar の MCP は接続済み。**Slack MCP（または Slack Web API）と LINE Bot の Webhook を足せば**、ハブはほぼ完成。
> - 入口を増やしすぎない。**LINE 1 本にまず絞る**のが続く秘訣。多経路化はデータ整理のコストを倍化させる。
